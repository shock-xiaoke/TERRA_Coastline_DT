Coastline Digital Twin and Prediction System Business Process Design

Core Principle: The system is driven by a dual-track system: a front-end "real-time inference loop" and a back-end "asynchronous assimilation loop." This avoids frequent model fine-tuning and instead achieves a near-real-time digital twin effect by updating the database state.

I. User Operation Flow

Initialization View: When the user enters the system interface, the map automatically loads the focused AOI (e.g., St. Andrews) and the latest real coastline status for that area.

Input Request: The user enters or selects the desired future time span for prediction in the interactive panel (e.g., entering 1 predicts 1 year later, entering 3 predicts 3 years later).

Result Acquisition and Visualization: The system responds very quickly, returning the "future coastline" generated by the prediction model as a vector line (GeoJSON) to the front-end, which is then overlaid and rendered on the AOI map, allowing the user to intuitively compare the current situation with future trends.

II. System Execution Flow
Besides handling user prediction requests and visualization rendering, the core function of the system execution flow is maintaining the vitality of the Digital Twin (DT) system. The entire system is divided into two completely decoupled subsystems: synchronous inference service and asynchronous data pipeline.

1. Synchronous Inference Service (Responding to User Clients)
When receiving a user request to "predict the next N years," the system executes the following steps:

State Fetching: The system extracts the latest distance sequence data for all coastlines of the AOI from the time-series database over a past period (e.g., the past 24 months).

Model Inference: This latest historical sequence is used as input features and fed into the trained prediction model. The model outputs the coastline N years from now.

Spatial Reconstruction: The system reverse-engineers the "one-dimensional discrete distance values" output by the model back into "two-dimensional geographic coordinate points" and connects them to form a smooth predicted coastline.

Response Encapsulation: Encapsulates the generated prediction line into standard GeoJSON format and returns it to the front end.

2. Asynchronous Data Assimilation Pipeline (Maintaining DT Vitality)

This is the system's backend "heartbeat," automatically executed periodically (e.g., triggered every 5 days) without user intervention to update the current state of the DT:

New Data Polling and Download: A scheduled task queries the Sentinel-2 API to check if the AOI has the latest satellite imagery. If the cloud cover meets quality requirements (e.g., < 30%), the imagery is downloaded; otherwise, the update is skipped.

Segmentation: The pre-trained segmentation model is used to infer the latest downloaded imagery and extract the latest coastline data.

State Update: After simple outlier cleaning, the latest distance data is appended to the time-series database.

Periodic Retraining: Set up a long-term mechanism (such as every 6 months) so that when the database has accumulated enough new real samples, the background will automatically trigger a full retraining of the prediction model to capture long-term landform evolution trends and replace the old prediction model weights.
